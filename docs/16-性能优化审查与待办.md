## 二、🔴 P0 严重问题

### P0-1. 三个 RAF 循环永不停转 —— CPU 占用头号元凶

**相关文件**
- `frontend/src/features/player/useAudioEngine.ts:77-176`（`loopSpectrum` / `tick`）
- `frontend/src/features/player/useBeatDetector.ts:41-141`
- `frontend/src/components/SpectrumPulse.tsx:31-55`

**问题代码**（useAudioEngine.ts，tick 末尾）

```ts
const tick = () => {
  // ... getByteFrequencyData + writeSpectrum + updateBeatCam ...
  usePlayerStore.setState({ beatCamEvents, beatCam }); // 160-171 行附近
  rafRef.current = requestAnimationFrame(tick);        // 172 行，无条件续帧
};
```

**根因**

- `useAudioEngine` 和 `useBeatDetector` 在 `App.tsx:84` **同时挂载**，构成**两套独立的 `requestAnimationFrame` 循环**并行运行。两者通过 `if (usePlayerStore.getState().beatmap)` 做了分工（有 beatmap 时 useAudioEngine 负责回放、useBeatDetector 退出），但**无 beatmap 时（云曲、本地未分析完）两个都在跑**。
- 三个 RAF 循环**完全不感知 `document.hidden`**。切到后台标签页或窗口最小化后，浏览器虽把 RAF 降频到 ~1fps，但每一帧仍跑 `setState` + 订阅回调，继续吃 CPU。
- `useBeatDetector` **不感知 `isPlaying`**：`!isPlaying` 时只是写 0 后 `return`（54-62 行），但 RAF 本身继续 `requestAnimationFrame`，即**暂停时仍每帧空转**。
- `SpectrumPulse` 挂在侧边栏（常驻），**不播放时也每帧 `readSpectrum()` + 写 5 个 CSS 变量**，触发 style invalidation。

> 说明：上一轮（12 号文档）已把 spectrum/beat 数据迁出 Zustand，消除了「setState → React 重渲染」放大效应。但**RAF 调度本身**这一层遗留未处理——数据流是零拷贝了，可循环还在每帧空跑。

**影响**：播放时主线程持续 ~60fps；后台/暂停时仍占用 CPU。这是项目里**最大的单一 CPU 消耗源**。

**修复方案**

方案 A（最小改动，保留双循环）：
```ts
// useAudioEngine.ts tick 开头
const tick = () => {
  if (document.hidden) {
    rafRef.current = requestAnimationFrame(tick); // 续帧但不干活
    return;
  }
  // ...原有逻辑...
};

// useBeatDetector.ts：!isPlaying 时 cancel，onPlay 时重启
useEffect(() => {
  if (!isPlaying) {
    cancelAnimationFrame(rafRef.current);
    rafRef.current = 0;
    return;
  }
  rafRef.current = requestAnimationFrame(tick);
  return () => cancelAnimationFrame(rafRef.current);
}, [isPlaying]);
```

方案 B（推荐，彻底）：合并成单一 RAF。由 `useAudioEngine` 驱动，把节拍检测函数作为回调注入：
```ts
// useAudioEngine.ts
const tick = () => {
  if (document.hidden || !isPlayingRef.current) {
    rafRef.current = requestAnimationFrame(tick);
    return;
  }
  const buf = readOrSampleSpectrum();
  writeSpectrum(buf);
  if (hasBeatmap) updateBeatCamFromBeatmap();
  else detectBeatRealtime(buf);   // 原 useBeatDetector 的逻辑内联
  rafRef.current = requestAnimationFrame(tick);
};
```

对 `SpectrumPulse`：用 `useVisibleRaf` + `enabled: isPlaying`，或 `!isPlaying` 时跳过 `setProperty`：
```ts
const tick = () => {
  if (isPlaying && !document.hidden) {
    const spectrum = readSpectrum();
    for (let i = 0; i < DOT_COUNT; i++) { /* setProperty */ }
  }
  raf = requestAnimationFrame(tick);
};
```

**验证**：播放时用 DevTools Performance 录制，确认后台标签页 RAF 回调内 setState/读 bus 不再执行；任务管理器观察后台标签页 CPU 归零。

### P0-2. `library.all()` 全库深拷贝在热路径反复触发

**相关文件**
- `crates/orange-library/src/database.rs:91-93`（`all()` 实现）
- `crates/orange-tauri/src/commands.rs:1526`（`recommend_next`）
- `crates/orange-tauri/src/commands.rs:1750-1754`（`analyze_track_bpm`）
- `crates/orange-library/src/database.rs:751-757`（`aggregate_user_profile`）
- `crates/orange-tauri/src/commands.rs:91-101`（`library_tracks` 无分页分支）

**问题代码**

```rust
// database.rs:91-93
pub fn all(&self) -> Vec<Track> {
    self.tracks.read().clone()   // 整份 Vec<Track> 深拷贝
}

// commands.rs:1750-1754 —— 仅为按 id 找一首歌，竟 clone 整库再 find
let bpm = state.library.all().into_iter().find(|t| t.id == track_id)
    .and_then(|t| t.bpm);
```

**根因**

`Track` 含 `TrackMeta`（多个 `String` + `Option<String>` + `Vec<String> genre` + `Option<Artwork>`），每首约几百字节到几 KB。`all()` 是 `self.tracks.read().clone()`——把整个内存库的 `Vec<Track>` **深拷贝一份**。万首库 ≈ 几十 MB。

这个 `all()` 在多个热路径被调用：
- `recommend_next`（懂你模式每次切歌）→ `aggregate_user_profile` 内部**又 clone 一次**进 `HashMap<String, Track>` → 一次推荐 = **2 次整库 clone**
- `analyze_track_bpm` 仅为按 id 找一首歌 → clone **整个库** 再 `find`
- `library_tracks` 无分页分支 → 进入「全部音乐」页就深拷贝整库

**影响**：万首库的推荐/分析路径，每次触发都分配几十 MB 临时内存 + GC。

**修复方案**（按改动量从小到大）

1. **`analyze_track_bpm` 直接改用已有的 `find`**（database.rs:101 已有此 API）：
```rust
// 改前：state.library.all().into_iter().find(|t| t.id == track_id)...
// 改后：
let bpm = state.library.find(&track_id).and_then(|t| t.bpm);
```
这一行改动，单点收益明确。

2. **`aggregate_user_profile` 的 `tracks_map` 只存必要字段**，不要 clone 整个 `Track`：
```rust
// 改前：HashMap<String, Track>（每首 clone 一整份）
// 改后：
let tracks_map: HashMap<String, (String /*artist*/, Vec<String> /*genre*/, Option<f64> /*bpm*/)> = {
    let guard = self.tracks.read();
    guard.iter().map(|t| (t.id.0.to_string(), (t.meta.artist.clone(), t.meta.genre.clone(), t.bpm)))
        .collect()
};
```

3. **`recommend_next` 候选池用共享只读快照**：`all()` 改为返回 `Arc<Vec<Track>>`（或新增 `all_arc()`），推荐引擎内部传 `&[Track]` + 索引，避免 `recommend.rs` 里 `t.clone()`（141/165/174/187 行）。

4. **`library_tracks` 强制走分页**：前端始终传 `offset+limit` 走 `query_paged`（已有 SQLite 实现，database.rs:190），无分页分支兜底也强制分页或加最大上限。

### P0-3. SQLite 连接不复用，每次操作重开 + 重复建表

**相关文件**：`crates/orange-library/src/database.rs` 全文，共 **20 处** `Connection::open`

**典型位置**：`set_liked`（509）、`update_track_bpm`（551）、`record_play_history`（642）、`recent_track_ids`（667）、`aggregate_user_profile`（731）、每个歌单操作（267/285/307/326/354/374/408）。多数还**重复调 `init_schema`**（如 222/249/268/327 行），每次 `CREATE TABLE IF NOT EXISTS` 全跑一遍。

**根因**：`set_liked`（点红心）、`record_play_history`（每首歌播完）、`aggregate_user_profile`（每次推荐）都是热路径，每次操作都 `Connection::open(path)` → `init_schema` → `prepare` → query → drop。无 prepared statement 缓存，每次重新解析 schema，文件句柄抖动。

**修复方案**

在 `LibraryDb` 持有一个长连接（rusqlite `Connection` 非 `Sync`，需 `Mutex` 包裹），或引入连接池：

```rust
// database.rs 结构体改造
pub struct LibraryDb {
    path: PathBuf,
    tracks: Arc<RwLock<Vec<Track>>>,
    conn: Arc<Mutex<Connection>>,   // 新增：长连接
    schema_init: OnceCell<()>,      // 新增：schema 只初始化一次
}

impl LibraryDb {
    pub fn open(path: impl AsRef<Path>) -> Result<Self, Error> {
        let conn = Connection::open(path.as_ref())?;
        conn.pragma_update(None, "journal_mode", "WAL")?;   // 顺带开 WAL
        conn.pragma_update(None, "synchronous", "NORMAL")?;
        let db = Self { path: path.as_ref().to_path_buf(), tracks: Arc::new(RwLock::new(Vec::new())), conn: Arc::new(Mutex::new(conn)), schema_init: OnceCell::new() };
        db.init_schema_once()?;
        Ok(db)
    }

    fn conn(&self) -> MutexGuard<'_, Connection> { self.conn.lock().unwrap() }

    fn init_schema_once(&self) -> Result<(), Error> {
        self.schema_init.get_or_try_init(|| {
            let conn = self.conn();
            // 原 init_schema 的 CREATE TABLE 语句
            Ok::<(), Error>(())
        })?;
        Ok(())
    }
}

// 每个方法改 self.conn() 取连接，去掉 Connection::open 和重复 init_schema
```

若不想引入 `r2d2`/`deadpool-sqlite` 依赖，单连接 + `Mutex` 对本项目（单进程、IPC 命令）足够。配合 `spawn_blocking`（已有），锁竞争极低。

> 注意：`LibraryDb` 当前是 `Clone`（只复制内部 `Arc`），新增 `Arc<Mutex<Connection>>` 字段不影响 `Clone` 语义。

### P0-4. QQ 音频流整曲缓冲进内存 —— 硬性内存峰值

**相关文件**：`apps/desktop/src-tauri/src/lib.rs:255-258`

**问题代码**

```rust
// handle_qq_stream
match upstream.bytes().await {
    Ok(bytes) => builder.body(Cow::Owned(bytes.to_vec()))   // 整首读进 Vec<u8>
}
```

**根因**：`handle_qq_stream` 每次请求都 `upstream.bytes().await` 把**整首歌**（Hi-Res/FLAC 可达 30-80MB，甚至 100MB）一次性读进 `Vec<u8>`，再以 `Cow::Owned` 返回。前端 `<audio>` 的 Range 请求虽透传了 `Range` 头，但 reqwest 默认不会因请求带 Range 就只读一段——`bytes()` 仍读到流结束。拖动进度条 / 快速切歌会触发多次全量缓冲。

对比 `handle_we_file`（272-384 行）是按 Range 精确 `read_exact`，QQ 流这里完全没做对等处理。

**影响**：QQ 音乐播放时 Rust 进程内存陡增几十 MB；拖进度条内存飙升；这是**唯一的硬性内存峰值风险**。

**修复方案**

优先：流式转发。Tauri 2 的 URI scheme responder 支持流式 body，避免整曲驻留：

```rust
use futures::StreamExt;

let upstream = client.get(&url).headers(range_headers.clone()).send().await?;
let status = upstream.status();
let headers = upstream.headers().clone();
let body_stream = upstream.bytes_stream().map(|chunk| {
    chunk.map(|c| c.to_vec()).map_err(|e| std::io::Error::new(std::io::ErrorKind::Other, e))
});
// 用 Tauri 2 的流式 Response 构造（具体 API 以 Tauri 2 文档为准）
builder.status(status).headers(headers).body(/* 流式 reader */)
```

退一步：拿到 `Content-Range`/`Content-Length` 后只 `bytes_range` 读取请求的区间。但仍是整段缓冲，不如流式彻底。

### P1-3. `COVER_CACHE` 无限增长

**相关文件**：`frontend/src/features/player/useCover.ts:13`

```ts
const COVER_CACHE = new Map<string, string | null>();  // 永不删除、无上限
```

**根因**：`proxyCoverUrl`（58-75 行）每次解析一张封面就 `COVER_CACHE.set(key, result)`，key 每首歌不同（`url:${url}` / `local:${path}` / `none:${id}`）。长时间听歌堆积成百上千条。

**修复方案**：换成 LRU，限 200 条。简易实现：
```ts
const COVER_CACHE = new Map<string, string | null>();
const COVER_CACHE_MAX = 200;

function cacheGet(key: string) {
  if (!COVER_CACHE.has(key)) return undefined;
  const v = COVER_CACHE.get(key);
  COVER_CACHE.delete(key); COVER_CACHE.set(key, v); // 提到最新
  return v;
}
function cacheSet(key: string, v: string | null) {
  if (COVER_CACHE.size >= COVER_CACHE_MAX) {
    COVER_CACHE.delete(COVER_CACHE.keys().next().value!); // 淘汰最旧
  }
  COVER_CACHE.set(key, v);
}
```

### P1-4. `bad_request` 用 `Box::leak` 泄漏内存

**相关文件**：`apps/desktop/src-tauri/src/lib.rs:272-273`

```rust
fn bad_request(msg: String) -> Response<Cow<'static, [u8]>> {
    let body: &'static [u8] = Box::leak(msg.into_boxed_str()).as_bytes(); // 永不释放
```

**根因**：每次 qqstream/wefile 出错（上游 5xx、路径越界）都 `Box::leak` 一段 `&'static` 字符串，永久泄漏。网络不稳时持续累积。

**修复方案**：改用 `Cow::Owned`，不必强求 `'static`：
```rust
fn bad_request(msg: String) -> Response<Cow<'static, [u8]>> {
    builder.status(400).body(Cow::Owned(msg.into_bytes()))
}
```

### P1-6. beatmap 分析双份等长 PCM（峰值 ~84MB）

**相关文件**
- `crates/orange-audio/src/decoder.rs:88-119`（`decode_file` 整曲 mono f32）
- `crates/orange-audio/src/beatmap.rs:110-136`（`biquad_lowpass` 又 `Vec::with_capacity(samples.len())` clone 一份等长）

**根因**：一首 4 分钟 44.1kHz mono ≈ 10.5M 采样 ≈ 42MB `Vec<f32>`；96kHz/24bit Hi-Res 更大。`biquad_lowpass` 再分配等长数组 → 峰值 ≈ 2×42MB = 84MB。`analyze_track_bpm` 和 `analyze_beatmap` 每次调用都产生这个峰值。

**修复方案**（按收益排序）
1. **BPM 分析只取前 30-60s**：节拍分析不需要全曲，截取前 ~60s 即可，内存直接降 4-10x。
2. **低通后下采样**：200Hz 低通后奈奎斯特允许降到 ~500Hz 采样率，onset 检测用下采样数据，数据量降 ~90x。
3. **`biquad_lowpass` 改原地**：写回输入 buffer，省一份 42MB。或节拍分析直接对分帧 RMS 做 biquad（帧数比采样数少 441 倍）。

### P1-7. `play_history` 清理触发条件坏掉，历史表实际从不清理

**相关文件**：`crates/orange-library/src/database.rs:653`

```rust
if now % 50 == 0 {   // now 是 Unix 秒
    prune_play_history(...);
}
```

**根因**：用 Unix 秒 `% 50 == 0` 触发清理——要求播放记录恰好落在「Unix 秒是 50 整数倍」的那一秒，概率极低且播放事件不是每秒都发。结果是 `play_history` 表**几乎从不清理**，持续增长（90 天/5000 条上限的清理逻辑本身是对的，但触发条件坏了）。

**修复方案**：改成计数器或时间戳触发：
```rust
// 方案：记录上次清理时间，每 24h 清一次
let last_prune = self.last_prune.load(Ordering::Relaxed);
if now - last_prune > 86400 {
    prune_play_history(...);
    self.last_prune.store(now, Ordering::Relaxed);
}
```
需在 `LibraryDb` 加 `last_prune: AtomicI64` 字段。

### P1-8. `set_liked` / `update_track_bpm` 走 JSON 读改写

**相关文件**：`crates/orange-library/src/database.rs:510-528`（set_liked）、`559-569`（update_track_bpm）

**根因**：tracks 表把整首 `Track` 存成一个 JSON `data` 列，改一个 `liked` bool 要：读整条 JSON → `serde_json::from_str` → 改字段 → `serde_json::to_string` → UPDATE 整条。`set_liked`（点红心，高频）还额外调 `add_to_playlist`（又一次 INSERT 整条 JSON）。

**修复方案**：把 `liked` 拆独立列（`ALTER TABLE tracks ADD COLUMN liked INTEGER DEFAULT 0`），`UPDATE tracks SET liked=? WHERE id=?`，避免 JSON 重写：
```rust
pub fn set_liked(&self, id: &str, liked: bool) -> Result<(), Error> {
    let conn = self.conn();
    conn.execute("UPDATE tracks SET liked = ?1 WHERE id = ?2", params![liked as i32, id])?;
    // 内存层同步
    let mut guard = self.tracks.write();
    if let Some(t) = guard.iter_mut().find(|t| t.id.0 == id) { t.liked = liked; }
    Ok(())
}
```
中长期把 `bpm`、播放次数等常用查询字段也提取为独立列。`query_paged` / `liked_tracks` 相应改用列查询而非 JSON 反序列化。

### P1-9. 聚合搜索并发无上限 + 超时不取消底层请求

**相关文件**：`crates/orange-tauri/src/commands.rs:1289-1314`（search_all）、`1557-1576`（recommend_next）

**根因**：9 个网络音源全部就绪时同时发起 9 路并发 HTTP；`recommend_next` 还补一轮（最多 2×9=18 个）。`tokio::time::timeout(5s)` 只让外层 future 在 5s 后返回 `vec![]`，但底层 `src.search` 的 reqwest 请求**不会被取消**，继续跑到自然结束或 client-level timeout，浪费带宽+CPU+连接池。

**修复方案**：用 `Semaphore` 限制并发 + `JoinSet` 在超时时 abort：
```rust
use tokio::sync::Semaphore;
let sem = Arc::new(Semaphore::new(4));
let mut set = JoinSet::new();
for src in sources {
    if !src.is_ready() { continue; }
    let permit = sem.clone().acquire_owned().await?;
    let q = query.clone();
    set.spawn(async move {
        let _p = permit; // 持有 permit 直到结束
        tokio::time::timeout(Duration::from_secs(5), src.search(&q)).await
    });
}
// 超时统一处理时 set.shutdown().abort() 取消所有未完成
```

### P1-10. `in_flight` 去重 map 只进不出

**相关文件**
- `crates/orange-sources/src/http_client.rs:17`（`in_flight: DashMap<String, Arc<Mutex<()>>>`，124-128 行 `entry().or_insert_with`）
- `crates/orange-tauri/src/lib.rs:88`（`CoverCache.in_flight`，commands.rs:1833-1837 使用）

**根因**：`cache` 有 `prune_loop` 清理，但 `in_flight` 这个并发去重 map **只进不出**——每个不同的 url+auth 组合永久留一个 `Arc<Mutex<()>>`。单条很小，但键是完整 URL，长期运行缓慢增长。

**修复方案**：`prune_cache` 时同步清掉 `in_flight` 里不在 cache 中的条目；CoverCache 下载完成后从 `in_flight` 移除该 key（结果已落盘，后续走磁盘命中分支）。

### P1-11. 日志默认 `info` + 双 layer，release 下浪费

**相关文件**：`apps/desktop/src-tauri/src/lib.rs:36-50`

**根因**：release 默认 `EnvFilter::new("info")` 偏啰嗦——`search_all` 每次打 info、每个音源注册一条 info、health check 都打 info。两条 layer（控制台 + 文件）意味着每条日志格式化写两次。控制台层在 `windows_subsystem = "windows"` 下其实无人看，纯属浪费。

```rust
#[cfg(debug_assertions)]
let console_layer = fmt::layer().with_target(true).with_filter(filter.clone());
let file_layer = fmt::layer().with_writer(file_writer).with_filter(filter);
// release 下不挂 console_layer
```

### P1-12. `prune_covers` 每次下载后同步全目录扫描

**相关文件**：`crates/orange-tauri/src/commands.rs:1877`、`1892-1923`

**根因**：每次 `cover_proxy` 下载完封面都调 `prune_covers(&cache_dir, 256MB, 0.2)`，做 `read_dir` + 每文件 `metadata` + `accessed` + 排序。首启批量加载歌单封面时连下几十张，每次都 O(n log n) 全量 stat。

**修复方案**：改成每 N 次下载或每 X 分钟清一次（计数器或时间戳节流）：
```rust
let count = self.cover_dl_count.fetch_add(1, Ordering::Relaxed);
if count % 32 == 0 { prune_covers(...); }
```

## 四、🟡 P2 进程级 / 配置建议

### P2-1. 透明窗口强制 GPU 合成层

### P2-3. dev profile opt-level = 0
- **文件**：`Cargo.toml`（根 workspace）
- **影响**：symphonia / reqwest / rusqlite 在 opt-level 0 下慢几倍，开发时易误判应用慢。
- **建议**：`[profile.dev.package."*"] opt-level = 1`，依赖按优化编译，本 crate 仍 opt-level 0 保留快编译。

### P2-4. index.html 同步加载 6 个 Google Fonts
- **文件**：`frontend/index.html:9-12`（含超大 CJK 字体 Noto Sans/Serif SC）
- **影响**：首屏阻塞；CJK 字体 woff2 子集仍数 MB，离线/弱网白屏。
- **建议**：CJK 字体本地打包或按需懒加载；只 preconnect 首屏实际用到的字体。

### P2-5. weapi RSA 大数未缓存
- **文件**：`crates/orange-sources/src/weapi.rs:55-73`
- **影响**：网易云几乎每个接口（搜索/歌单/歌词/收藏/心跳）都调 `weapi_post` → 每次 `BigUint::parse_bytes` + `modpow`。MODULUS/PUB_KEY 是常量，可缓存。
- **建议**：`OnceLock::new()` 缓存 `e`、`n` 的 `BigUint` 解析结果；`PRESET_KEY` 的 `Aes128::new` 同理。

### P2-6. 网易云登录轮询 1.2s 打一次登录接口
- **文件**：`crates/orange-tauri/src/commands.rs:413-458`
- **影响**：登录窗口打开期间（用户几十秒才扫码）每 1.2s 都 `login_with_cookie` 打网易云接口。
- **建议**：先本地判断 cookie 是否含 `MUSIC_U`（有才发登录请求）；轮询间隔拉到 2-3s。

## 八、第二批实施记录（2026-07-10）

第二批「高收益 + 中等改动」已实施并通过构建双绿验证。以下记录实际改动、与原始方案的差异及未实施项的原因。

### 已实施

#### P0-3 SQLite 连接池化（database.rs 重构）
- **改动**：`LibraryDb` 新增 `conn: Option<Arc<Mutex<Connection>>>` 长连接字段 + `conn()` 辅助方法；全部 17 处 `Connection::open(path)` + 重复 `init_schema` 替换为复用 `self.conn()`；`open` 时一次性开启 WAL + synchronous=NORMAL 提升并发读写性能。
- **收益**：所有 DB 命令（set_liked / record_playback / recommend_next / 歌单操作等）不再重开连接、不重复建表，prepared statement 缓存生效。配合调用方 `spawn_blocking`，`parking_lot::Mutex` 锁竞争极低。
- **注意**：`db_path` 字段在连接池化后不再被读取（clippy dead_code），保留并加 `#[allow(dead_code)]` 注释用于诊断。

#### P0-2 扩展：aggregate_user_profile 轻量化
- **改动**：`aggregate_user_profile` 内的 `tracks_map` 从 `HashMap<String, Track>`（每首 clone 整个 Track，含 Artwork/album/source_track_id 等大字段）改为 `HashMap<String, TrackFeatures>`，只存 `(artist: String, genre: Vec<String>, bpm: Option<f32>)` 三元组。
- **收益**：懂你模式每次推荐的用户画像聚合，内存分配从「整库 Track 深拷贝」降到「每首 3 个字段」。
- **实现说明**：原方案试图用借用 `&str`/`&[String]` 零拷贝，但 `guard`（RwLock 读锁）生命周期无法覆盖后续统计循环，编译期借用检查失败。最终采用 owned 三元组（仍比 clone 整个 Track 省很多）。

#### P1-6 beatmap 下采样
- **改动**：
  - `biquad_lowpass` 改为 `biquad_lowpass_in_place`（原地写回输入 buffer，省一份等长 ~42MB Vec）。
  - 低通后下采样到 ~1000Hz（200Hz 低通后奈奎斯特 400Hz，1000Hz 足够），数据量降 ~44x（44100→1000），后续帧化/onset 内存与 CPU 大幅下降。
  - `analyze` 函数签名从 `&DecodedAudio` 改为 `&mut DecodedAudio`（调用方 `decode_file` 后立即分析，audio 用完即弃，原地安全）。
- **收益**：整曲节拍分析峰值内存从 ~84MB（PCM + biquad 副本）降到 ~1MB（下采样后），CPU 耗时同步下降。

#### P1-1 / P1-2 前端 store 订阅治理
- **改动**：
  - `WallpaperBackground.tsx`：删除 raw `usePlayerStore.subscribe((s) => ...)` 全量订阅，改由 effect 依赖 `[currentTrack]` 自然驱动切歌封面更新（原 subscribe 监听整个 store，每帧 beatCam/position setState 都触发回调）。
  - `FullPlayer.tsx` ProgressSection：音符动画的 raw subscribe 全量订阅改为 `useEffect([isPlaying])` 驱动；`visualParams` 整对象订阅拆成字段级（`lyricColor` / `lyricColorAuto` / `preset` 三个独立 selector）。
  - `WallpaperLayer.tsx`：对象 selector 加 `useShallow`，避免每次返回新对象字面量导致 store 任意变化都重渲染。
- **收益**：消除每帧 60 次的无效订阅回调；VisualConsole 拖滑块时重渲染范围收窄。

### 未实施（含原因）

#### P0-4 qqstream 流式转发 —— 因 Tauri API 约束放弃
- **原始方案**：把 `handle_qq_stream` 的 `upstream.bytes().await`（整曲读进内存）改为流式转发。
- **实际约束**：核查 Tauri 2.11 源码发现 `UriSchemeResponder::respond` 签名为 `pub fn respond<T: Into<Cow<'static, [u8]>>>(self, response: http::Response<T>)` —— **body 必须是完整的 `Cow<'static, [u8]>`，不支持流式**。在当前 Tauri 版本下无法真正流式转发。
- **现状评估**：现有实现透传了 Range 头。当上游 CDN 支持 Range（返回 206）时，`bytes()` 只读取请求的那一段，**并非总是整曲灌入内存**——内存峰值主要发生在上游不支持 Range（返回 200 全量）的边缘情况。
- **后续方向**：若要彻底解决，需升级到支持流式 body 的 Tauri 版本，或改用独立本地 HTTP 代理服务（如之前的 127.0.0.1 axum 方案）替代 scheme handler。
