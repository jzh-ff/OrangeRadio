# 01 - OrangeRadio 技术架构

> 版本：v0.2 · 最后更新：2026-07-01

## 一、产品定位

OrangeRadio 是一款基于 **Tauri 2 + Rust** 的**全平台沉浸式智能音乐播放器 + AI 音乐创作工作站**，目标全方位超越 Mineradio。详见 [08-产品规划](08-产品规划.md)。

## 三、当前实际架构（v0.2 已落地部分）

### 音频播放架构（重点：与初版规划的差异）

> ⚠️ **重要决策变更**：初版规划用 Rust symphonia 做 Hi-Res 解码，但 v0.2 实际采用了**前端 Web Audio API 播放方案**。原因与权衡见下方"关键技术决策"。

**实际播放链路（v0.2）：**
```
用户双击曲目
  → 前端 engineRef.playTrack(track)
  → useAudioEngine.playPath(filePath)
  → convertFileSrc(filePath)  // 转成 http://asset.localhost/<path>
  → <audio>.src = url
  → <audio>.play()           // 浏览器原生解码播放（有声音）
  → 播放状态写入 playerStore（position/duration/isPlaying）
  → 模拟频谱 → playerStore.spectrum → Three.js 渲染
```

**本地库扫描链路（v0.2）：**
```
用户点"扫描本地音乐"
  → 弹窗选目录（@tauri-apps/plugin-dialog）
  → invoke("library_scan", { rootDirs })
  → Rust: LibraryScanner.scan()
    → walkdir 递归遍历
    → lofty 读取每个音频文件的元数据（标题/艺术家/时长/封面）
    → 构建 Track 列表
  → LibraryDb.replace_all()  // 内存索引
  → 返回曲目列表给前端
  → 前端 libraryStore.setTracks() + playerStore.setQueue()
```

## 四、关键技术决策（记录踩过的坑）

### 决策 1：Windows 工具链用 GNU (WinLibs) 而非 MSVC

**背景**：机器无 MSVC Build Tools，winget 安装反复失败。
**方案**：安装 WinLibs MinGW-w64 (POSIX+MSVCRT, GCC 16) + Rust `stable-x86_64-pc-windows-gnu`。
**坑**：必须把 WinLibs bin 加到 PATH **最前面**，否则 Rust 会找到 LLVM-MinGW（clang-based，无 libgcc）导致链接失败。`run.ps1` 已自动处理。
**后续**：正式发布前建议切回 MSVC（Tauri 官方推荐）。

### 决策 2：vite 不设 base（默认 `/`）

**背景**：初版设了 `base: "./"`，应用启动白屏。
**根因**：Tauri 2 在 Windows 用 `http://tauri.localhost/`，ESM 模块的相对路径 `./assets/x.js` 解析失败。
**方案**：移除 `base`，用 vite 默认 `/`（生成绝对路径 `/assets/x.js`）。

### 决策 3：播放用纯 `<audio>`，不接 MediaElementSource

**背景**：为做频谱分析接了 `createMediaElementSource`，结果完全静音。
**根因**：`asset.localhost` 与 `tauri.localhost` 不同源，Web Audio CORS 安全机制把音频判为 tainted，输出全零。且接入 MediaElementSource 后原 `<audio>` 不再直接发声。
**方案**：只用 `<audio>` 元素播放（保证声音）。频谱用模拟数据驱动视觉，后续接 Rust 端 FFT 取真实频谱。

### 决策 4：本地库用内存索引（暂不用 SQLite）

**背景**：v0.2 聚焦"能播"，避免引入数据库复杂度。
**方案**：`LibraryDb` 用 `Arc<RwLock<Vec<Track>>>` 内存存储。每次启动重新扫描。
**后续**：v0.3 迁移 SQLite 持久化。
