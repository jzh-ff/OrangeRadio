# 12 · 性能优化记录（卡死修复 + 全面清理）

> 版本：v0.3.x · 分支：`feature/cinema-redesign`
> 日期：2026-07-06
> 关联：`docs/01-技术架构.md`、`docs/11-播放队列优化.md`

本文档记录一次针对「界面卡死/无响应」问题的系统性性能优化，包含根因分析、改动清单与验证方法，供后续维护参考。

---

## 一、问题现象与根因

### 1.1 现象
- 用户反馈「最近很容易卡死」，表现为界面冻结、点击无响应，需重启应用。
- 日志（`.orangeradio/logs/orangeradio.log.2026-07-05`）在 `17:34:28` 出现：
  ```
  ERROR tauri_runtime_wry: WebView2 error: WindowsError(Error {
    code: HRESULT(0x8007139F),
    message: "组或资源的状态不是执行请求操作的正确状态。"
  })
  ```
  此后日志中断 **39 分钟**，直到 `18:13:28` 才出现「启动中…」（用户手动重启）。
- `0x8007139F`（`GROUP_OR_RESOURCE_NOT_IN_CORRECT_STATE`）是 WebView2 在窗口/渲染线程挂起或资源回收时抛出，**指向渲染层（前端）无响应，而非 Rust 主线程 panic**（日志无 `panicked` 字样）。

### 1.2 根因（P0，决定性因素）
频谱可视化数据流设计错误，导致主线程被 React 渲染 + GC 占满：

| 环节 | 文件 | 问题 |
|---|---|---|
| 写入 | `useAudioEngine.ts` `loopSpectrum` | 在 `requestAnimationFrame`（~60fps）里每帧 `usePlayerStore.setState({ spectrum: Array.from(data) })`，每帧 `new Uint8Array` + `Array.from` 分配新数组写 Zustand |
| 放大 | `HeroSpectrum.tsx` / `RightWaveFlow.tsx` | `useEffect` 依赖数组含 `spectrum` → 每帧 `cancelAnimationFrame` + `ResizeObserver.disconnect` → 重建 RAF + ResizeObserver |
| 连锁 | `SpectrumPulse.tsx` | 订阅 `spectrum`，每帧重渲染写 5 个 CSS 变量 |
| 节拍 | `useBeatDetector.ts` | 每帧 `setBeat(...)` 再写一次 store |

**结果**：每秒 ~120 次 store 写入 → 多个组件每帧重渲染 + 两个 canvas 的 RAF 循环每帧「拆-建」→ 主线程饱和 → WebView2 无响应。

### 1.3 次要因素（非卡死主因，但会放大卡顿）
- **后端**：async Tauri 命令直接同步调用 `rusqlite`（`LibraryDb` 全套方法），无 `spawn_blocking`。慢盘/大库时占满 tokio worker，连带 `ping`/UI 心跳排队。
- **后端**：网易云 / QQ / Spotify / 网络电台 / 播客 5 个音源的 `reqwest::Client` **无 timeout**。对端不响应时单源命令无限挂起、占死 worker。
- **前端**：`LibraryView` 用 `useLibraryStore()` 整体订阅 store；`VirtualTrackList` 的 `Row` 是内联组件未 memo → 虚拟化收益被抵消。
- **前端**：`PlaylistShelf` 的 `CanvasTexture` 切换/卸载不 dispose（显存泄漏）；`useAudioEngine` 的 `AudioContext`/`AnalyserNode`/`MediaStreamSource` 卸载不 disconnect/close（Web Audio 泄漏）；`FullPlayer` 一个 `requestAnimationFrame` 句柄未保存无法 cancel。

## 二、优化措施

### 2.1 P0：频谱/节拍数据迁移到模块级 bus（核心修复）

**思路**：把每帧变化的高频数据移出 Zustand store，改走「模块级同步 bus」——复用同一份 `Uint8Array` / 对象，零分配、零 GC、零 React 重渲染。消费方（canvas / Three.js / 节拍检测）在各自的 RAF / `useFrame` 里主动 `readSpectrum()`。

**新增** `frontend/src/stores/spectrumBus.ts`：
- `writeSpectrum(Uint8Array)` / `readSpectrum(): Uint8Array`
- `writeBeat(Partial<BeatState>)` / `readBeat(): BeatState`
- `resetSpectrumBus()`（切歌/卸载时清残留）

**改写**：
| 文件 | 改动 |
|---|---|
| `useAudioEngine.ts` | 删除每帧 `setState({ spectrum })`，改 `writeSpectrum(buf)`；beatmap 回放的 `setState({ beat })` 改 `writeBeat(...)`，仅 `beatmapIndex`（低频）留 store |
| `useBeatDetector.ts` | `getState().spectrum` → `readSpectrum()`；`setBeat(...)` → `writeBeat(...)` |
| `HeroSpectrum.tsx` | 删 `usePlayerStore((s) => s.spectrum)` 订阅；effect 依赖 `[isPlaying, bars]`（去掉 spectrum）；draw 内 `readSpectrum()` |
| `RightWaveFlow.tsx` | 同上 |
| `SpectrumPulse.tsx` | 去掉 store 订阅，改用 `useEffect` 起轻量 RAF 从 bus 读数据写 CSS 变量 |
| `VisualBackground.tsx` | 两处 `useFrame` 内 `getState().spectrum` → `readSpectrum()` |
| `playerStore.ts` | `spectrum` / `beat` 字段加 `@deprecated` 注释（保留兼容，初值不动） |

**效果**：主线程每帧不再触发任何 React 重渲染；HeroSpectrum/RightWaveFlow 的 effect 只在 `isPlaying`/`bars` 变化时执行一次。GC 压力归零。

### 2.2 P1：大列表订阅泄漏
- `LibraryView.tsx`：`useLibraryStore()` 整体订阅 → `useShallow` 细粒度选 9 个字段，store 任意字段变化不再触发整组件重渲染。
- `TrackRow.tsx`：`TrackRow` 加 `memo`；`VirtualTrackList` 采用 react-window 官方推荐的 `itemData` + 模块级 `MemoRow` 模式，把依赖打包 `useMemo`，引用稳定时所有可见行不重渲。

### 2.3 后端：SQLite 调用包 `spawn_blocking`
`crates/orange-tauri/src/commands.rs` 中 11 个命令的同步 `rusqlite` 调用全部包进 `tokio::task::spawn_blocking`（`LibraryDb` 是 `Clone`，内部全 `Arc`，安全 move）：
- `library_scan`（`replace_all` 的 `persist_local`）
- `toggle_liked` / `create_playlist` / `rename_playlist` / `delete_playlist` / `add_to_playlist` / `remove_from_playlist`
- `playlist_tracks` / `all_playlists`
- `record_playback` / `get_user_profile`
- `recommend_next`（把 `aggregate_user_profile` + `recent_track_ids` + `recent_feedback` + `all` 打包成一次 `spawn_blocking`，`recommender` 的 async 算法在外层 await）

**模式**：
```rust
let library = state.library.clone();
tokio::task::spawn_blocking(move || library.xxx(...))
    .await
    .map_err(|e| e.to_string())?   // JoinError
    .map_err(|e| e.to_string())    // 业务 Error
```

### 2.4 后端：音源 reqwest client 加 timeout
统一参考 `gequbao` 的 15s：
| 音源 | 文件 | timeout |
|---|---|---|
| 网易云（主 client） | `netease.rs` | 15s |
| 网易云（扫码 qr_client） | `netease.rs` | 10s |
| QQ 音乐（主 client） | `qqmusic.rs` | 15s |
| QQ 音乐（扫码 session） | `qqmusic.rs` | 10s |
| Spotify | `spotify.rs` | 15s |
| 网络电台 | `web_radio.rs` | 15s |
| 播客 | `podcast.rs` | 15s |

> 注：`search_all` 聚合搜索本身已有 `tokio::time::timeout(5s)` 兜底；本次补的是各**单源命令**（`netease_search`/`qqmusic_stream`/`radio_search` 等）的缺省超时。

### 2.5 内存泄漏清理
- `PlaylistShelf.tsx`：`placeholderTex` / `tex` / `coverTex` 各加 `useEffect(() => () => x.dispose(), [x])`，切换/卸载时释放 GPU 纹理。
- `useAudioEngine.ts`：保存 `MediaStreamSource` 节点引用；卸载 cleanup 里 `srcNodeRef.disconnect()` + `analyserRef.disconnect()` + `ctxRef.current.close()` + `resetSpectrumBus()`。
- `FullPlayer.tsx`：歌词居中的补帧 `requestAnimationFrame` 句柄存 `lyricCenterRafRef`，`useLayoutEffect` 返回 cleanup `cancelAnimationFrame`。

### 2.6 未改动（说明）
- `LyricOverlay.tsx` 的 250ms `position` 定时器：经核实跑在**独立歌词悬浮窗 webview**（无 `<audio>` 元素），是该窗口估算播放进度的必要机制，与主窗口 `timeupdate` 不冲突；改动会引入歌词漂移风险，保留。

## 四、后续性能监测建议

1. **临时开 debug 日志**：`RUST_LOG=debug` 环境变量启动，可看到每次 IPC 命令的耗时与 spawn_blocking 调度。
2. **加 panic hook**（建议后续在 `apps/desktop/src-tauri/src/lib.rs` 的 `run()` 里）：
   ```rust
   let default_hook = std::panic::take_hook();
   std::panic::set_hook(Box::new(move |info| {
       tracing::error!(target: "panic", "PANIC: {info}");
       default_hook(info);
   }));
   ```
   这样若后端真 panic，会在日志留痕，便于区分「渲染层卡死」与「后端 panic」。
3. **前端性能分析**：Chrome DevTools（Tauri debug 构建）→ Performance 录制，重点看 RAF 回调耗时与 React commit 频次。优化后主线程在播放时应基本空闲（频谱绘制走 canvas 直绘，不进 React）。
4. **大库压测**：放入 5000+ 首本地音乐，观察扫描耗时、搜索响应、滚动 FPS。若仍卡，下一步可考虑 `LibraryDb::search` 建内存索引（当前是 O(n) 三字段 `to_lowercase` 全表扫）。
